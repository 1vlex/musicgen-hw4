# Report

## Краткое описание

В этой работе был реализован полный пайплайн fine-tuning `MusicGen` на `MusicCaps` с использованием структурированных метаданных. Исходные текстовые caption были преобразованы в JSON фиксированной схемы, затем эти поля были подмешаны в conditioning prompt и использованы для обучения `musicgen-small`.

## Подготовка данных

Для сбора аудио был использован датасет `google/MusicCaps`. Аудио скачивалось не целиком, а как 10-секундный фрагмент через связку `yt-dlp + ffmpeg`.

Схема работы была такой:

- `yt-dlp` получал прямую ссылку на аудиопоток
- `ffmpeg` скачивал только нужный 10-секундный отрезок в `.wav`
- рядом с каждым `.wav` сохранялся `.source.json` с исходной строкой датасета

После этого caption преобразовывались в structured JSON с обязательными полями:

- `description`
- `general_mood`
- `genre_tags`
- `lead_instrument`
- `accompaniment`
- `tempo_and_rhythm`
- `vocal_presence`
- `production_quality`

## Использованная LLM и системный промпт

Для обогащения метаданных использовался Groq-compatible API с моделью `openai/gpt-oss-20b`.

Лучше всего сработал консервативный системный промпт из `prompts/metadata_system_prompt.txt`. В нем были зафиксированы такие правила:

- вернуть только один JSON-объект
- использовать строго заданные ключи
- не добавлять лишние поля
- не галлюцинировать лишние детали
- если поле неясно, использовать `Unspecified`
- если вокала нет, использовать `None`

Такой промпт оказался полезен, потому что уменьшал число шумных метаданных и делал structured sidecar более стабильным.

## Изменения в AudioCraft

В локальном клоне `AudioCraft` были внесены изменения, чтобы модель могла использовать новые structured поля.

Сделано было следующее:

- расширен `MusicInfo`
- добавлено чтение structured JSON sidecar
- добавлена логика сериализации structured полей в единый текстовый prompt

В этой реализации structured поля не передавались как независимые conditioners. Вместо этого они склеивались в один текстовый prompt и подавались в `description`. Такой вариант оказался проще и надежнее для `musicgen-small` в условиях ограниченной памяти.

## Гиперпараметры обучения

Использовалась модель `facebook/musicgen-small`.

Финальные параметры обучения:

- batch size: 4
- learning rate: `5e-5`
- epochs: 10
- updates per epoch: 62
- segment duration: 10 секунд
- warmup: 50 steps
- early stopping patience: 4
- early stopping min epochs: 4
- early stopping min delta: `1e-4`
- `merge_text_p = 0.85`
- `drop_desc_p = 0.20`
- `drop_other_p = 0.80`
- classifier free guidance training dropout: `0.3`

Размер выборки:

- train: 248 клипов
- valid: 10 клипов

## Результаты обучения по логам

После исправления конфигурации с warmup обучение стало работать корректно и завершилось по early stopping на 8 эпохе.

По логам:

- лучшая `valid ce` была достигнута на эпохе 1: `3.546306`
- последняя `valid ce` на эпохе 8: `3.546335`
- early stopping сработал на эпохе 8

Некоторые значения из финального запуска:

- Epoch 1: `train ce = 4.038`, `valid ce = 3.546306`, `duration = 250.571 s`
- Epoch 2: `train ce = 4.079`, `valid ce = 3.546352`, `duration = 334.820 s`
- Epoch 3: `train ce = 4.092`, `valid ce = 3.546354`, `duration = 342.996 s`
- Epoch 8: `train ce = 4.128`, `valid ce = 3.546335`, `duration = 171.537 s`

По метрикам видно, что улучшение на валидации было очень небольшим. Скорее всего, модель быстро вышла на плато. При этом по субъективному прослушиванию итоговые 5 тестовых генераций получились достаточно похожими на заданные промпты по жанру, настроению и ведущим инструментам.

## Качество генерации

Качественная оценка проводилась по 5 тестовым структурированным промптам.

По прослушиванию:

- жанр и общее настроение в большинстве случаев угадываются
- ведущие инструменты обычно распознаются достаточно хорошо
- точное соблюдение всех полей промпта не всегда идеальное
- качество аудио в целом приемлемое, без грубых артефактов

Итоговая субъективная оценка: генерации получились достаточно похожими на целевые описания и подходят для демонстрации результата fine-tuning.

## Трудности

Главные сложности в работе были такими:

- сбор данных занял много времени, потому что выгрузка аудио из YouTube идет медленно и не всегда стабильно
- часть ссылок требовала повторных попыток, поэтому пришлось делать ретраи в скрипте скачивания
- при обогащении метаданных LLM периодически возвращала ошибку `429`, что ограничивало скорость разметки и напрямую влияло на объем данных, который удалось подготовить за разумное время
- на Windows возникли проблемы с сохранением и перезаписью чекпоинтов, поэтому пришлось дополнительно чинить локальный пайплайн сохранения
- после перехода на PyTorch 2.6 понадобилось отдельно исправить загрузку training checkpoints из-за `weights_only=True` по умолчанию

Еще одна важная проблема состояла в том, что датасет получился небольшим. Для full fine-tuning это чувствительно. Из-за этого модель довольно быстро вышла на плато по валидационной метрике.

## Почему не LoRA

Для такого объема данных LoRA могла бы быть более практичным вариантом:

- меньше вычислительная стоимость
- проще хранить веса
- ниже риск нестабильности на маленьком датасете

Тем не менее в задании просили именно fine-tuning `MusicGen`, поэтому в работе использовался обычный fine-tuning `musicgen-small`, без перехода на LoRA.

В результате был собран рабочий пайплайн:

- скачивание аудио из MusicCaps
- structured metadata через LLM
