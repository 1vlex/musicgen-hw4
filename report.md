# Report

## С какими трудностями пришлось столкнуться

Основная сложность была в подготовке данных.

- выгрузка аудио из YouTube занимает много времени и работает не всегда стабильно
- часть ссылок приходилось обрабатывать с повторными попытками
- при обогащении метаданных LLM периодически возвращала ошибку `429`, что ограничивало скорость разметки
- на Windows возникли проблемы с сохранением и перезаписью чекпоинтов во время обучения
- после перехода на PyTorch 2.6 пришлось отдельно исправлять загрузку training checkpoints из-за `weights_only=True` по умолчанию

Еще одна проблема связана с размером датасета. Для full fine-tuning `musicgen-small` выборка получилась небольшой, поэтому обучение довольно быстро вышло на плато.

## Какую LLM использовал и какой системный промпт сработал лучше

Для построения structured metadata использовался Groq-compatible API с моделью `openai/gpt-oss-20b`.

Лучше всего сработал консервативный системный промпт из `prompts/metadata_system_prompt.txt`. Его идея была такой:

- вернуть только один JSON-объект
- использовать строго заданные ключи
- не добавлять лишние поля
- не галлюцинировать лишние детали
- если поле неясно, использовать `Unspecified`
- если вокала нет, использовать `None`

Такой подход дал более стабильные sidecar JSON и снизил количество шумных аннотаций.

## Какие гиперпараметры обучения использовались

Использовалась модель `facebook/musicgen-small`.

Основные параметры:

- batch size: 4
- learning rate: `5e-5`
- epochs: 10
- updates per epoch: 62
- segment duration: 10 секунд
- warmup: 50 steps
- early stopping patience: 4
- early stopping min epochs: 4
- early stopping min delta: `1e-4`
- `merge_text_p = 0.85`
- `drop_desc_p = 0.20`
- `drop_other_p = 0.80`
- classifier free guidance training dropout: `0.3`

Размер выборки:

- train: 248 клипов
- valid: 10 клипов

## Что получилось по логам обучения

После исправления конфигурации warmup обучение стало идти корректно и завершилось по early stopping на 8 эпохе.

По логам:

- лучшая `valid ce` была достигнута на эпохе 1: `3.546306`
- последняя `valid ce` на эпохе 8: `3.546335`
- early stopping сработал на эпохе 8

Ключевые значения:

- Epoch 1: `train ce = 4.038`, `valid ce = 3.546306`, `duration = 250.571 s`
- Epoch 2: `train ce = 4.079`, `valid ce = 3.546352`, `duration = 334.820 s`
- Epoch 3: `train ce = 4.092`, `valid ce = 3.546354`, `duration = 342.996 s`
- Epoch 8: `train ce = 4.128`, `valid ce = 3.546335`, `duration = 171.537 s`

По метрикам улучшение на валидации оказалось очень небольшим. Скорее всего, модель быстро вышла на плато из-за малого объема данных.

## Оценка качества генерации

Итоговые 5 аудио по субъективному прослушиванию получились достаточно похожими на заданные structured prompts.

Что было заметно:

- жанр и общее настроение в большинстве примеров передаются убедительно
- ведущие инструменты и общий характер аранжировки хорошо считываются на слух
- итоговые 10-секундные фрагменты звучат цельно и без грубых артефактов
- в субъективном прослушивании генерации получились достаточно близкими к заданным structured prompts

В целом, по субъективной оценке, итоговая генерация получилась удачной и достаточно близкой к требованиям задания.

## Почему не LoRA

Для такого объема данных LoRA могла бы быть более удобным вариантом:

- меньше вычислительная стоимость
- проще хранить веса
- ниже риск нестабильности на маленьком датасете

Тем не менее в задании просили именно fine-tuning `MusicGen`, поэтому в работе использовался обычный fine-tuning `musicgen-small`, без перехода на LoRA.

## Вывод

В результате был собран полный рабочий пайплайн:

- скачивание аудио из MusicCaps
- structured metadata через LLM
- подготовка manifests
- модификация AudioCraft
- fine-tuning `musicgen-small`
- инференс по 5 тестовым structured prompts

Несмотря на небольшой датасет и слабое улучшение по `valid ce`, итоговые генерации по субъективной оценке получились достаточно близкими к целевым описаниям.
